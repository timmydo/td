summary: Task 2a — give the modern /td/store toolchain (gcc-14.3.0 + binutils-2.44 + glibc-2.41, gate 412) a STABLE KEY, the prereq for td-subst chain-caching (2b/2c). The toolchain is not byte-reproducible (cc1 stamp, ar/install mtimes), so `store-add-recursive`'s content-addressed `/td/store/<nar-hash>-<name>` path VARIES build-to-build — a td-subst consumer can't name what to fetch. This expresses the toolchain INPUT-ADDRESSED instead: `tests/td-toolchain.lock` declares its complete pinned input set (24 sources + 7 patches + 3 components), `td-builder toolchain-key` hashes those (order-independent), and `td-builder toolchain-path LOCK [NAME]` prints `/td/store/<key-digest>-<name>` — a pure function of the inputs, so it is identical across non-reproducible rebuilds and computable from the lock BEFORE fetching. New placement primitive `td-builder store-add-input-addressed NAME KEY SRC STORE-DIR OUT-DB` (mirrors store-add-recursive but the path digest is KEY, not the NAR hash; the placed tree still registers its REAL NAR hash for closure/verify). New HEAVY gate `toolchain-input-addressed` (414), durable + td-native end to end (no guix oracle): pinned-sync (lock pins == seed/sources + seed/patches), stable-key (deterministic, distinct component paths), content-indep (same key+different bytes -> same path, while content-addressed splits), load-bearing (a pin moves the path), behavioral+structural (a real binary at an input-addressed path runs in the store-ns own-root, /gnu/store absent). Producer wiring of the literal toolchain (swap gate 412's glibc-2.41 intern to store-add-input-addressed) is 2b's opening step (where the ~90-min build is inherent), not this PR. Builder/src validates on cargo-test + check-engine.
