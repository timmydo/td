summary: Stand up td's OWN builder daemon so the loop realizes derivations without guix-daemon. Increment 1 (landed, #69): `td-builder realize` computes the build's input closure ITSELF (td's SQLite reader over the store db's Refs graph, replacing `guix gc -R`), builds in its userns sandbox, registers the output. Increment 5 (landed, #112): make the build sandbox SELF-hermetic — `sandbox::build` pivot_roots into a minimal root (staged /gnu/store + /tmp + /dev & /proc rbind'd + minimal /etc), so a build can no longer reach the host filesystem even when invoked OUTSIDE the outer host-sandbox; durable gate `build-hermetic` (356) proves a realized probe cannot see /var/guix. Increment 6 (landed, #126): full pid-namespace parity — `sandbox::build` unshares NEWPID and forks the builder to PID 1 of its own pid namespace with a FRESH procfs, so a build sees only its own process tree, not the host's; folded into the build-hermetic gate (356). Increment 7 (this claim): the PERSISTENT BUILD DAEMON — `td-builder daemon SOCKET STORE-DB SCRATCH` is a long-running process that realizes derivations served over a Unix socket (the track's end goal: the loop's builder instead of guix-daemon), wrapping the exact `realize_drv` path; `td-builder daemon-request SOCKET DRV` is the in-process client. Durable gate `build-daemon` (358): one daemon serves TWO distinct realize requests over one socket (persistence), each producing td's OWN output (under the scratch store, not /gnu/store) with the expected marker, daemon-free. Increment 8 (this claim): the daemon realizes a TD-ASSEMBLED real-recipe `.drv` into td's OWN store — the maintainer's 2026-06-21 direction (closing #128: shipping goes through td's OWN daemon/store, NOT the guix daemon) done right. New `td-builder assemble-recipe` (assemble-only: td assembles the recipe `.drv` with `store::assemble_drv` — no Guile, no realize), so assembly and realization split across processes; the persistent daemon then realizes that td-assembled drv (default guix-built builder ⇒ every input-src daemon-valid ⇒ realize over /var/guix/db, no overrides) and CACHES a valid output (guix-daemon parity: a valid path is not rebuilt). Durable gate `daemon-recipe` (359): the daemon builds a real artifact into td's store from a td-assembled drv — NO `guix repl`/Guile in the daemon's input path, output under td's scratch store (not /gnu/store), the artifact runs, a 2nd request for the same drv is a cache HIT. Next: heavier Rust userland (cat/coreutils) through this exact path; the loop consuming the daemon by default.
