summary: td's OWN substitute (binary-cache) server — the dual of td-feed. Where td-feed mirrors SOURCE downloads (content-addressed, self-verifying), td-subst serves BUILT /td/store outputs as NAR + signed metadata so a consumer (td install / CI image prep / a cold worktree) fetches a path instead of REBUILDING it. A new Rust binary subst/ modeled on feed/ (std::net + already-vendored ring for ed25519 + sha2): `publish DB STORE OUTDIR PATH` writes nar/<narhash>.nar (nar::write_nar) + a signed td-native <pathhash>.narinfo (path|narhash|narsize|refs from store-query/store-closure); `serve OUTDIR ADDR` answers GET /<hash>.narinfo + /nar/<…> index-free with verify-on-serve; `selftest` does a loopback publish→serve→fetch→read_nar-unpack→byte-identical round-trip with self-discrimination legs (corrupt NAR, bad signature, tampered hash all red). Protocol = td-native minimal (human 2026-06-24), trust = ed25519-signed narinfo verified against a pinned td public key WITH a durable repro-equality gate leg (fetched-into-clean-store == from-source rebuild, byte-identical) + tool-runs behavioral leg; removable migration-oracle leg = td-subst /nar bytes == `guix archive --export PATH`. Consumer wired into td-builder: before build_and_register, compute the output path, query the server, and if a verified substitute exists fetch+read_nar+store-register instead of building — OFF for the verification loop (directive 1: the loop always builds from source + --check), opt-in for td install/CI/host-warm. New primitive: nar::read_nar (the inverse of write_nar; the format is fully specified in nar.rs's header). Inc1 = read_nar + verified-red round-trip; Inc2 = subst binary publish/serve/selftest loopback; Inc3 = ed25519 signing + pinned-key verify + self-discrimination; Inc4 = td-builder consumer hook + the durable fetch-instead-of-build gate.
