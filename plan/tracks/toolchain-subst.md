summary: Tasks 2b + 2c — register the /td/store toolchain to td-subst and let a consumer FETCH it by the key 2a gave it ([[td-toolchain-input-addressed]]). The td-subst gate (358) already proves the full publish->sign->serve->fetch->consumer-hook round-trip for CONTENT-addressed paths; this adds the genuinely new lock-keyed leg: a real artifact is interned at the INPUT-ADDRESSED path `td-builder toolchain-path tests/td-toolchain.lock glibc-2.41` via store-add-input-addressed, subst-export'd + signed + served, and a CONSUMER that has ONLY the lock independently computes the SAME /td/store path (asserted equal), fetches it by that basename, verifies the ed25519 signature + the narinfo StorePath == its lock-computed path + NarHash, restores it, and RUNS the fetched-not-built binary. Trust is the SIGNATURE + the input-addressed name, NOT repro-equality (the toolchain is not byte-reproducible — that is task 3). Self-discrimination: a wrong public key reds the fetch. This is the "skip the rebuild" payoff for an input-addressed artifact, end to end and signature-trusted. The literal gcc/glibc bytes flow through the IDENTICAL path; wiring gate 412 to intern its toolchain components input-addressed + subst-export them is the producer step that runs in the DAILY heavy suite (a ~90-min from-seed build, infeasible per-PR / with cold sources), so it is documented, not wired here. Reuses 358's already-built subst binary + keys (no new heavy build). Validates by running the td-subst gate (builds subst from source offline — its crate FODs are warm in /gnu/store).
