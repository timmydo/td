# Track: m12 — signed distribution (mainline)

**Claim status:** see `PLAN.md` (the single source of truth for claims).
**Origin:** mainline successor to M11; blocked on M11.
**Scope authority:** DESIGN §7.1 (approved roadmap — no further sign-off needed).

## Goal

A generation image is pushed to and pulled from a registry (local/offline inside
the loop), its signature verified before placement; the placer rejects unsigned
or tampered images.

## Acceptance (DESIGN §7.1)

Push, pull, verify-then-place green end-to-end inside the offline loop; the
placer rejects unsigned and tampered images (verified-red).

## Pre-decisions (DESIGN §2.7 — settled 2026-06-10, on paper, do not re-litigate)

- **Identity = digest of the distributed artifact** in its canonical form
  (today: sha256 of the docker-archive tarball; the OCI manifest digest once
  the `oci-load` track lands the canonical layout — a `DIGESTS.md`
  re-baseline). Signatures are over the digest, never the install ordinal N;
  "verify before placement" means digest first, slot second.
- **Signing vehicle: detached ed25519 signature over the digest; no sigstore**
  (cosign is Go-heavy and network-assuming — likely the next podman).
  Candidates: signify/minisign-style tool, or guile-gcrypt's ed25519 (already
  a build dependency of `system/td-generation.scm`). State the M8-style probe
  criterion here BEFORE adopting: offline-buildable from the pinned channel,
  sane derivation count.
- **Registry: a static layout, not a product.** A static `oci-layout`
  directory plus a trivial local server (or plain file transport) IS the
  registry inside the loop; no registry product enters the loop.
- **Downgrade policy: anti-rollback is out of scope.** Reject exactly three
  things: unsigned, bad signature, digest mismatch. An old but validly signed
  generation is re-placeable by design (manual rollback is the model).

## Open question (decide here when the track starts — §2.7/§6)

- **Key distribution**: how the target gets the verifying public key — placer
  flag, well-known path on `td-state`, or baked into the placer build.

## Carried-over to-do (from M10.3 — §2.7)

- **Placed `image-digest=` line**: the placer additionally writes
  `image-digest=sha256:…` into `boot/td/gen-N/td-identity`, computed over the
  artifact it actually unpacked. M10.3 landed WITHOUT this one-line increment,
  so it falls to M12's start; it is M12's verification anchor (verify the
  signature over the digest, hash the pulled artifact, compare, only then
  place — the placed record then states the verified identity).

## Constraints

- Offline loop, as always (FSDG relaxed to a non-goal 2026-06-11 — DESIGN §5);
  the probe-before-adopt discipline above is the hard gate on any new tool.
- Placer changes strengthen `place-check`; new rungs touch shared-spine
  `Makefile`/`check.sh`: standalone commits (§7.3).

## Working state

**Claimed 2026-06-11 (claude-fable-c4148a), draft PR #7.**

### Decisions at track start

- **Key distribution (the named open question): placer flag (`--pubkey FILE`).**
  Explicit and hermetic — the test passes the key like any other input; no hidden
  machine state. Rejected: baked-into-placer (the placer is a plain POSIX script;
  embedding key material couples rotation to a placer change) and td-state
  well-known path (invents a §2.6 allowlist entry before any deployment needs
  it — can layer on later as a convention for WHERE the flag's file comes from,
  without changing the placer contract).
- **Digest today = sha256 of the docker-archive tarball** (§2.7 staging; the
  placer hashes the artifact it actually unpacked). Moving identity to the OCI
  manifest digest when distribution adopts the oci-load canonical layout is a
  recorded representation change (DIGESTS.md re-baseline), not a convention
  change — revisit at S3.

### Sub-task ladder

- **S1 — placed `image-digest=` line (the §2.7 carried-over anchor).** The placer
  appends `image-digest=sha256:<hex>` (computed over `--image`) to the placed
  `boot/td/gen-N/td-identity`. Test first: `place-check` asserts the line is a
  well-formed sha256 and — when the rung passes the artifact paths (new
  `TD_IMAGES` env, wired in the `place` rung via `place-drv` printing the image
  store paths) — that it EQUALS the artifact's actual sha256. Verified-red:
  (A) assertion before implementation (line missing), (B) placer defect hashing
  the wrong content (manifest.json instead of the artifact) caught by the
  equality leg.
- **S2 — signing vehicle probe (M8-style, criterion before adoption).**
  Candidates: `signify` / `minisign` from the pinned channel (placer-side verify
  needs a tool on the placer's PATH — same pattern as veritysetup for --mkfs);
  guile-gcrypt ed25519 for the SIGNING side if convenient (already a dep).
  Criterion: offline-buildable from the pinned channel, sane derivation count,
  usable from the guix-free placer sandbox. Probe, decide, record here.
- **S3 — push/pull through a static registry layout.** Push = populate a static
  layout (file transport; no registry product) with the artifact + detached
  signature over its digest; pull = retrieve by digest; pulled artifact
  byte-identical to pushed. Decide here whether the layout is the oci-load
  canonical `oci-layout` (then identity = manifest digest, DIGESTS.md
  re-baseline) or the docker-archive + sidecar signature.
- **S4 — verify-then-place.** The placer (--pubkey) verifies the detached
  signature over the digest, hashes the pulled artifact, compares, and only then
  places; rejects exactly: unsigned, bad signature, digest mismatch
  (verified-red ×3 — the §7.1 acceptance). Anti-rollback explicitly out of
  scope (§2.7).
