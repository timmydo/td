# Track: m10 — bootc-style generations (mainline)

**Claim status:** see `PLAN.md` (the single source of truth for claims).
M10.1–M10.3 DONE (records in `HISTORY.md`; M10.3 landed 2026-06-11). Next on this
thread: M11 — split into its own track file when started.
**Scope authority:** DESIGN §7.1 (approved roadmap — no further sign-off needed).
**Full design:** `M10-design.md`.

A generation is a bootc-style bootable OCI image; you build it with Guix, place it,
add a GRUB entry, and roll back by booting an older entry — essentially Guix's own
generation menu. Re-scoped 2026-06-08 to the simplest useful form: **manual rollback**.
No boot counters, no health agent, no auto-commit.

## Standing decisions (from the scoping discussion — still hold)

1. **State boundary: define, don't abandon.** "Nothing persists across test runs" is
   test-isolation (fresh disk per test), not a ban on persistence *within* a test. M10
   introduces guest-persistent state (the generation list / boot selection) for the
   first time; in the manual model it's small (a GRUB menu + extracted roots).
2. **Oracle = Guix builds the bundle** (reproducible, `--check`). The deployment side
   (place + menu update) is tested against behavior, not diffed against a Guix
   component it doesn't have.
3. **Integrity ≠ security.** Corruption detection is ordinary functional behavior;
   authenticity, signatures, and anti-rollback are M12 (signed distribution).

## M10.3 — manual rollback + declared persistence (next)

Acceptance (DESIGN §7.1): from a disk carrying two placed generations, the marionette
test boots generation N, asserts its identity (root label / system), reboots selecting
generation N−1 from the GRUB menu, and asserts the older identity. Persistence is
asserted per the DESIGN §2.6 state model (settled 2026-06-10 — it governs over any
older wording in this file), in both directions: a declared `td-state` allowlist path
written under generation N persists into the N−1 boot; an undeclared write does not
follow the swap. M10.3 therefore also introduces the `td-state` filesystem, the
declared-persistence allowlist bind-mounts, and SSH host-key relocation (§2.6).

Open follow-ons M10.2 left for M10.3:

- Turn each staged `roots/td/gen-N/root.tar` into a per-generation **read-only ext4
  image** LABELED `td-root-gen-N` (`mke2fs -d -L`, reproducible UUID/hash_seed) —
  ext4 as a read-only image container (M11's dm-verity data device), not a
  traditional rw filesystem; `td-state` stays the only one of those (§2.6).
- The menuentry path is GRUB-root-relative `/td/gen-N/...` and root selection is by
  label — wiring the GRUB `search`/`set root` for the target partition layout,
  `set default`, and the gnu.system/gnu.load closure path is M10.3's job.
- Target-persistent state across boots (the first time the loop reboots a guest
  mid-test rather than resetting it): the `td-state` filesystem and the §2.6
  allowlist mounts, asserted both directions as above.

## Working state

### M10.3 design (settled 2026-06-10 after reading the pinned Guix sources)

**Boot contract discovered.** The Guix initrd mounts the `root=` device from the
kernel command line, then `primitive-load`s the file named by `gnu.load` (no
`gnu.load` = no boot, it drops to a REPL); `/run/current-system` is set by the
activation from `gnu.system` on the command line (`boot-time-system`). So the
placer's menuentry must carry `root=<gen-label> gnu.system=<system>
gnu.load=<system>/boot` — the menu wiring IS the bootable-or-not lever, which makes
the identity assertion (`readlink /run/current-system` == that generation's system
path) a direct test of the placer.

**Containerization problem + decision.** td's stock OCI lowering
(`system-docker-image`) wraps the OS in `containerized-operating-system`: dummy
kernel, getty/networking services stripped, file-systems removed. The staged root
from M10.2 therefore carries a *container-flavored* system closure — its boot script
is not honestly bootable on a disk (no real kernel modules, no declared mounts).
Decision: the generation bundle's userspace layer is repacked from the REAL
(non-containerized) OS closure via `build-docker-image` over
`initialize-root-partition` — exactly what the qcow2 root partition contains, as an
OCI layer. The plain containerized docker image stays what it is (the M5/M8 OCI
artifact and the DRV_BASE no-/boot discriminator); the *bundle* is the bootable one.

**Identity binding.** `boot/td-identity` gains `system=<system store path>` (what the
menu must point at — it must exist inside the userspace layer) and
`root-uuid=<operating-system-uuid os 'dce>` (deterministic per-OS; the placer uses it
for the reproducible mkfs). The placer verifies generation/root-label as before and
now *reads* system/root-uuid instead of taking new trust inputs.

**Placer growth (still guix-free).** `td-place.sh`:
  * menu entries get `--id td-gen-N` + the gnu.system/gnu.load args from identity;
  * managed block gains `search --no-floppy --label <boot-label> --set=root` (only
    when `--boot-label` is given), `set default=td-gen-<newest>`, and a rollback
    hook: `if [ -s /td/default.cfg ]; then source /td/default.cfg; fi` — manual
    rollback = write `set default=td-gen-<N-1>` into that file on the boot
    partition and reboot; the placer never writes it;
  * `--mkfs`: extract the staged root.tar and `mke2fs -t ext4 -d <dir> -L <label>
    -U <root-uuid> -E root_owner=0:0,...` (the exact invocation Guix's own
    `make-ext-image` uses — the proven-reproducible path) → `roots/td/gen-N/root.img`,
    sized like `estimate-partition-size` (du + 25%). The place DERIVATION wraps the
    placer in `fakeroot` (as Guix does for mke2fs); on a real target you are root.
  * optional `--extra-kernel-args` (test disk adds `console=ttyS0`).
  The PLACE/PRUNE rung scenarios stay mkfs-less (cheap); the rollback tree uses
  `#:mkfs? #t`.

**Disk assembly (test fixture, NOT claimed guix-free — the target's disk exists; the
placer is the guix-free deliverable).** New `system/td-disk.scm`: partition 1 = ext4
label `td-boot`, fixed UUID, bootable, content = placed tree's `boot/` + the GRUB
i386-pc modules (what grub-install would have copied); partitions 2,3 = the placer's
root.img files verbatim; assembled with genimage (mbr, 1 MiB first-partition offset =
GRUB's core.img gap); GRUB installed image-side like Guix's
`install-grub-disk-image`, except prefix `(hd0,msdos1)/grub` (dedicated boot
partition, paths GRUB-root-relative — matching the placer's `/td/gen-N/...` lines).
Raw output; `guix build --check` proves the whole disk reproducible.

**The rollback test (tests/rollback.scm, rung `rollback`).** Generations 1+2 built
with `#:transform-os` = marionette instrumentation (same overlay-style as the
disk-boot test; the keyword defaults to identity so the other rungs' bundles are
untouched). The test builder creates ONE qcow2 overlay backed by the store disk and
boots it TWICE (install-test pattern; `marionette-pid` + waitpid between sessions,
distinct socket dirs):
  * session 1 (GRUB default = newest): assert identity — /proc/cmdline carries
    root=LABEL=td-root-gen-2, `stat:dev /` == `stat:rdev /dev/disk/by-label/...`,
    readlink /run/current-system == gen-2's system path; write a persistence
    sentinel onto the boot partition; write `set default=td-gen-1` to
    /td/default.cfg (the manual rollback act); clean `reboot` (QEMU has -no-reboot
    → exits).
  * session 2 (same overlay): assert gen-1 identity the same three ways; assert the
    sentinel survived and gen-2's menu entry + placed files are still present
    (placed state persists across the reboot — the acceptance clause).

### S5 — the §2.6 state model (scope landed by the human 2026-06-10, mid-flight)

DESIGN §2.6 + the M10-design.md state-model section landed on main while S1–S4
were in flight; caught at the pre-landing rebase, absorbed as S5 (it is binding,
pre-approved M10.3 scope). Implementation:

- **Typed config**: `persistent-paths` field — list of
  `(precious|disposable . "/abs/path")` — default `((precious . "/var/lib/ssh"))`.
  Generation-mode only (§2.6 oracle scope): with generation #f nothing is
  emitted (typed-diff still converges); the coverage rung wires it via a
  structural row (drv-divergence is impossible for a deliberately-inert-by-
  default field) and validates shape + the cross-field rule that a generation
  system must keep the precious /var/lib/ssh entry (machine identity).
- **Compiler emission (generation set)**: td-state fs (LABEL=td-state at
  /td-state, needed-for-boot → the INITRD mounts it); one bind-mount
  file-system per entry (device = /td-state/<state|cache><path>), mounted by
  shepherd — `user-processes` requires `file-systems`, so every daemon starts
  after the binds. No `dependencies` on td-state (a needed-for-boot fs has no
  shepherd service; the initrd phase IS the ordering).
- **Ordering insight**: activation runs after the initrd's mounts but before
  shepherd's — so the host-key mint writes THROUGH THE BACKING PATH
  (/td-state/state/var/lib/ssh/...), not the bind; sshd later reads it through
  the bind. (Bind sources are not root-prefixed by the initrd's
  mount-file-system, so the binds themselves cannot be needed-for-boot.)
- **SSH identity**: sshd_config gains `HostKey /var/lib/ssh/ssh_host_ed25519_key`
  via extra-content; generate-host-keys? #f (no per-generation /etc/ssh noise);
  activation mints the ed25519 key on first boot if missing.
- **Harness**: td-disk.scm adds the td-state partition (64 MiB, fixed UUID,
  tier roots + default backing dirs pre-created; the GUEST writes the actual
  state — that is the point).
- **Test**: per boot, /var/lib/ssh must be BACKED by td-state (st_dev equals
  /td-state's, not /'s); a declared-path sentinel written under gen 2 must
  survive into gen 1 while an undeclared write must not; gen 2's minted host
  key must be byte-identical under gen 1.

### Second --check-red finding (mke2fs nondeterminism, 2026-06-10)

The first rollback-rung run went red at `guix build --check` on the mkfs tree:
bare `mke2fs` stamps the superblock/journal with the current time and a random
directory-hash seed. td's qcow2 `build` rung never exposed this because partition
images are INPUTS of the derivation `--check` re-runs (the mke2fs step is never
re-executed); the placer derivation embeds mkfs directly, so the loop caught it.
Fix: `SOURCE_DATE_EPOCH=1 E2FSPROGS_FAKE_TIME=1` + `-E hash_seed=<uuid>` + pin the
extracted top-dir mtime (placer) / normalize all mtimes (td-disk bootroot, which
copy-recursively re-stamps). Verified deterministic by two host runs (bare differs
at byte 1073; pinned invocation bit-identical) and by the rung's --check going
green.
