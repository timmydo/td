# Track: m11 — verified generations (mainline)

**Claim status:** see `PLAN.md` (the single source of truth for claims).
**Origin:** mainline successor to M10.3 (`plan/m10.md`); unblocked — M10.3 is
DONE (status: `PLAN.md`; record: `HISTORY.md`).
**Scope authority:** DESIGN §7.1 (approved roadmap — no further sign-off needed).

## Goal

A generation's root carries build-time integrity metadata; booting an intact
generation succeeds while a corrupted root fails closed. Integrity ≠ authenticity:
signatures are M12.

## Acceptance (DESIGN §7.1)

Booting an intact generation succeeds; a corrupted root fails closed —
verified-red by corrupting bytes in the placed root image.

## Settled decisions (consolidated from DESIGN — do not re-litigate)

- **Mechanism (settled 2026-06-10): dm-verity over the per-generation root
  image**, ChromeOS-style. `veritysetup format` (FIXED salt — reproducibility)
  emits a hash tree + root hash at build time; the root hash rides the kernel
  cmdline in the GRUB menuentry. M10.3's per-generation read-only ext4 image
  becomes the verity data device UNCHANGED. (§7.1)
- fs-verity was rejected (per-file only — cannot verify a root); composefs is
  re-parked in DESIGN §6 (would replace, not extend, the per-generation-image
  design; not in the pinned Guix). (§7.1)
- **The tmpfs-root assembly lands together with M11's sealing** (§2.6): at boot
  `/` is tmpfs, the generation image is mounted read-only, activation
  materializes `/etc`, `/run`, `/tmp` — a root the boot path writes to cannot be
  sealed. M11 turns read-only-by-convention into kernel-enforced (undeclared
  writes fail closed, EROFS) — the §2.6 enforcement stage after M10.3's
  convention stage.
- **Verification boundary, stated honestly** (§7.1): at boot only the root below
  `/boot` is verified — kernel, initrd, and the cmdline carrying the hash are
  trusted as placed, not re-verified. Placement-time authenticity is M12; a
  signed boot chain is off-roadmap.

## Constraints

- Probe before adopting (the M8 lesson): `veritysetup` / cryptsetup and the
  kernel dm-verity target must be offline-buildable from the pinned channel
  with a sane derivation count — record the probe numbers here before wiring
  anything in.
- The hash tree build must be reproducible (fixed salt; `guix build --check`
  stays the oracle).
- New/changed rungs touch shared-spine `Makefile`/`check.sh`: standalone
  commits (§7.3). Strengthening the rollback/boot rungs is free; weakening
  anything needs the human (§4.3(2)).

### S0 — probes (2026-06-11, claude-fable-7d8371; M8 lesson: probe before adopting)

**Kernel (pinned linux-libre 6.18.15, the built store output's `.config`):**
- `CONFIG_DM_VERITY=m` — module present
  (`lib/modules/6.18.15-gnu/kernel/drivers/md/dm-verity.ko.zst`, dep
  `dm-bufio.ko.zst` alongside); `CONFIG_BLK_DEV_DM=y`, `CONFIG_CRYPTO_SHA256=y`.
- `CONFIG_DM_INIT is not set` — the ChromeOS-style `dm-mod.create=` cmdline
  path is OUT (no kernel rebuild on the roadmap). The verity device is opened
  by USERSPACE veritysetup in the initrd — Guix's own mapped-device pattern
  (LUKS-root precedent, behavior-proven).
- Consequence: generation initrd-modules += `dm-verity` (boot-system loads
  modules BEFORE pre-mount, where mapped devices open — order is right).

**veritysetup (pinned channel):**
- `cryptsetup-static` (the initrd variant LUKS uses) is built with
  `--disable-veritysetup` — it canNOT open verity devices. M11 defines a
  td-local static variant (inherit cryptsetup-static; drop that flag; keep
  `sbin/veritysetup` instead of `sbin/cryptsetup` in the remove-cruft phase).
- Build cost: `guix build cryptsetup-static -n --no-substitutes` = 17 drvs
  (static popt/json-c/argon2/libgcrypt/libgpg-error + lvm2-static + cryptsetup
  itself) — sane, offline-buildable. The td variant shares all the static-lib
  inputs.
- Place-time tool: the full `cryptsetup` package ships `veritysetup` (dynamic
  is fine inside the place derivation; on a real target it is a placer
  runtime requirement like mke2fs).

**Pinned initrd machinery (read 520785e sources):**
- `boot-system` (gnu build linux-boot): mounts essential fs -> loads modules ->
  `pre-mount` (raw-initrd splices each mapped-device kind's `open` gexp here)
  -> mounts root fs -> mounts remaining needed-for-boot fs under /root ->
  switch-root -> primitive-loads gnu.load.
- `root=` on the cmdline OVERRIDES the declared root device; with NO `root=`
  arg the declared "/" file-system governs. `canonicalize-device-spec` passes
  any non-`/dev/` string (e.g. tmpfs's "none") through unresolved — a
  DECLARED tmpfs root mounts fine with no cmdline root= at all.
- `mapped-device-kind`: `open` = (source targets -> gexp), `modules` = extra
  Guile modules for the initrd gexp. The open gexp may read the cmdline
  ((gnu build linux-boot) linux-command-line/find-long-option) and resolve
  labels ((gnu build file-systems) find-partition-by-label).
- needed-for-boot binds: sources are NOT root-prefixed in the initrd (M10.3
  finding, still true) — so the store CANNOT be a needed-for-boot bind of a
  side-mounted full-tree image. Drives the image-content decision below.

### Settled implementation shape (from the probes + §2.6/§2.7)

**Per-generation artifact (placer `--mkfs`, ChromeOS append style):** one
`root.img` = ext4 data area (size pinned to a 4 KiB multiple) + verity hash
tree appended at `--hash-offset` = data size. `veritysetup format` with FIXED
salt and `--uuid` = the identity's deterministic root-uuid → bit-reproducible
(the rollback-drv rung's `--check` is the oracle). Placer records
`verity-roothash` + `verity-hashoffset` next to root-label/system/root-uuid;
the menuentry carries `td.roothash=<hash> td.hashoffset=<bytes>` — the hash
rides the cmdline (§7.1), placed by the placer, because the image cannot
carry its own root hash (§2.7 self-reference, same reason as the digest).

**Boot (generation mode only — generation #f stays byte-identical to the
oracle):** `/` = declared tmpfs (no root= on the cmdline); the verity device
`/dev/mapper/td-root` opens in pre-mount via a td mapped-device kind (static
veritysetup; data dev resolved by the generation's root LABEL — still the
slot-binding mechanism; roothash/hashoffset parsed from the cmdline) and
mounts READ-ONLY at `/gnu/store`, needed-for-boot, fsck off (dm-verity IS the
integrity check; fsck would try to write a ro device). Activation
materializes /etc, /run, /tmp on the tmpfs root (§2.6 "the root is assembled,
not stored"). td-state + allowlist binds unchanged from M10.3.

**Image content consequence (the one §7.1 nuance, recorded honestly):** the
ext4-as-verity-data-device design is unchanged, but the image's fs root
becomes the `gnu/store` SUBTREE of the staged root.tar (mke2fs -d
$fsroot/gnu/store), not the full tree: with a tmpfs `/`, the store is the
only thing the image still provides at runtime (plus the system closure in
it), and the initrd cannot needed-for-boot-bind a subdir of a side mount
(probe note above). The OCI artifact (root.tar — what M12 signs) is
untouched; the placed td-identity `system=` check still runs against
root.tar.

**Fail-closed semantics:** a corrupted block read returns EIO (dm-verity
default); corruption in the boot path = mount/exec failure, observable as
the kernel's `device-mapper: verity ... is corrupted` signature on the
serial console. Undeclared writes: store/system → EROFS (kernel-enforced,
the §2.6 enforcement stage); tmpfs `/` → absorbed and gone on reboot
(strictly stronger than M10.3's "lingers in the generation root").

### Assertion-reshape inventory (§4.3 discipline — none of these are
weakenings; each existing assertion maps to an equal-or-stronger replacement
forced by the §2.6-settled tmpfs-root + verity boot shape; counts only go up)

| existing assertion | M11 replacement |
|---|---|
| rollback: cmdline `root=<label>` selects gen-N (not gen-M) | cmdline `td.roothash=<gen-N's recorded hash>` (not gen-M's) — cryptographic, not just a label string |
| rollback: `/` IS the fs labeled td-root-gen-N | `/gnu/store` IS `/dev/mapper/td-root`, whose backing slave is the partition labeled td-root-gen-N; PLUS `/` is tmpfs; PLUS the store mount is ro |
| place-check (3): entry selects own `root=<label>`, no foreign label | entry carries own `td.roothash=`/`td.hashoffset=` (values from that gen's recorded files), no foreign roothash, and NO `root=` arg at all |
| place-check (1c) superblock label/UUID | unchanged (data area still starts at byte 0) + NEW verity-superblock magic at the recorded hash offset |

Everything else (system path identity, td-state backing, declared/undeclared
sentinels, host-key identity, menu structure, prune, preamble) is untouched.
